## 1. Hearts — new assets + threshold fix — `screens/lesson/LessonSessionScreen.tsx`

- Swapped the ❤️/🖤 emoji + `assets/map/half-heart.png` for three new assets
  the user added: `assets/map/redh.png` (full), `halfh.png` (half),
  `whiteh.png` (empty).
- Found and fixed a real bug while doing this: the visual (5 icons × 2
  half-heart states = 10 total capacity) didn't match the actual
  session-ending trigger, which fired at 5 mistakes, not 10 — so the "Out of
  Hearts" modal could appear while the icons still visually showed ~2.5
  hearts left. Hoisted `MAX_HEARTS = 5` / `MAX_MISTAKES = MAX_HEARTS * 2` as
  shared module constants so the two can't drift apart again.
- `assets/map/half-heart.png` is now unused but was **not deleted** (out of
  scope for the asset-cleanup pass — flagged to the user, pending their call).

## 2. Play/pause — real icons + state-order bug fix — `screens/lesson/LessonSessionScreen.tsx`, new `components/PlayPauseIcon.tsx`

- No dedicated play/pause icon image assets exist in the project, so built a
  small shared SVG icon (`react-native-svg`, already a dependency) — a
  triangle/bars icon-swap, used instead of the ▶/⏸ emoji glyphs (which
  render inconsistently across Android OEM emoji fonts).
- Wired into `PlayPauseBtn`, `SegmentPlayBtn`, `AyahDisplay`'s big circular
  button, and `HintButton`'s modal button — the four "first two ayahs of
  each level" play buttons (per the user's explicit scope).
- Fixed a real bug in all four at the same time: each one called
  `setState(...)` **after** `await playUrl(...)`, but `playUrl` doesn't
  resolve until playback *finishes* — so the icon showed the wrong state
  (▶ during playback, ⏸ right as it ended) for the entire duration of every
  play. Reordered to set state before awaiting in each handler.
- Left `AudioFill`'s per-option play circles and `HearAndSelect`'s speaker
  button untouched (out of the requested "first two ayahs" scope, still
  emoji-based — noted as a possible follow-up).

## 3. Season-gate deadlock fix — `screens/home/MapScreen.tsx`

- `isSeasonComplete()` checks whether the previous season's last surah's
  last level is `completed` — correct logic (sequential progression means
  this implies everything earlier is done too) — but it only had data to
  check when that surah happened to still be in `fullLevels`, which stops
  being true once the "current surah" pointer moves on to the new season.
  Gates could get permanently stuck saying "Finish Season N to unlock" even
  after the user had actually finished it.
- Fixed by fetching that one surah's levels on demand the moment the gate is
  tapped (if not already cached), with a "Checking…" loading state on the
  Lumo speech bubble while it resolves.

## 4. Real Leaderboard — `screens/leaderboard/LeaderboardScreen.tsx`, `api/index.ts`, `types/api.ts`

- Backend shipped `GET /api/v1/leaderboard` (top 13 users by XP,
  `{rank, display_name, xp}`, authenticated, 60s server-side cache).
- Replaced the fully-mocked screen (hardcoded 10 fake players, a fake "You"
  always at rank 4) with a real fetch, loading/error/empty states, and
  pull-to-refresh.
- **Known limitation, by design for now:** the endpoint has no `user_id` or
  `is_me` flag, so "you" is highlighted by matching `display_name` against
  the logged-in user's own name. This misfires if two users share a display
  name, and if the user isn't in the top 13 they won't be highlighted
  anywhere in the list. Flagged to the user; can be revisited if backend
  adds an `is_me` field or a separate "my rank" endpoint.

## 11. Confetti timer leak fix — `screens/lesson/LessonSessionScreen.tsx`

- A bare `setTimeout(() => setShowConfetti(false), 1500)` could fire after
  the screen unmounted (e.g. lesson ends within 1.5s of a correct answer),
  calling `setState` on a dead component. Tracked the timer in a ref and
  clear it on unmount / before starting a new one.

## 12. Dead code removed — `store/lessonStore.ts`, `screens/lesson/LessonSessionScreen.tsx`

- Removed `recordAttempt` and `nextStep` from `lessonStore` — confirmed via

  removed the now-unused `exerciseTypeForApi` import and
  `isEndedSessionError` helper (only existed to support `recordAttempt`),

## 14. Backend follow-up round — `api/formatError.ts`, `screens/lesson/LessonSummaryScreen.tsx`, `screens/profile/ProfileScreen.tsx`

Backend made `/learning/sessions/{id}/complete` server-authoritative (grades
from real formula-engine attempt data instead of trusting client-sent
`passed`/`score_pct`/`mistakes`) and removed `POST /progress/xp` (was an
open XP-cheating vector — confirmed via grep the mobile app never called it,
no action needed). Two things did need frontend changes:

- Added a friendly message for HTTP 429 (new backend rate-limiting on
  login/verify-email/reset-password) in `messageForStatus` — previously fell
  through to a raw `Request failed (429)`.
- `hearts_remaining` can now be fractional (e.g. 9.5, since 1 mistake = half
  a heart server-side). Grepped every display site — two render the raw
  number as plain text next to a heart icon
  (`LessonSummaryScreen.tsx`'s stat card, `ProfileScreen.tsx`'s stat card).
  Wrapped both in `Math.round(...)`. The in-lesson heart icons (§1 above)
  are unaffected — they're driven by the session's own local mistake
  counter, not this backend field.
